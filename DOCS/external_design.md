## 2. Screen Design

### 2.1 Main Screen (Job Input / Result Display)

**Layout**

- Page title: `IT Engineer & Job Requirement Matching (RAG Demo)`
- Subtitle: Brief description of the system (1-2 lines)

**Input Area**

- Label: "Enter job requirements"
- Component: `st.text_area` (approx. 400px height)
- Placeholder: Sample input text displayed as a hint
- Button: "Run Matching" (`st.button`)

**Result Display Area**

- Displayed after search execution (with loading spinner)
- Each candidate engineer displayed using `st.expander`
- Display fields: rank, engineer name, skill summary, similarity score, matching rationale
- Rationale displayed as bullet points (2-3 items)

### 2.2 Sidebar (Index Build)

**Layout**

- Header: "Admin Menu"
- Description text: Brief explanation of index build (1 line)
- Button: "Build Index" (`st.button`)
- Post-execution display: Success message (record count) or error message
- ChromaDB status display: Current record count (`st.metric`)

## 3. Input/Output Definitions

### 3.1 Input (Job Requirement Text)

| Field | Detail |
|-------|--------|
| Field Name | Job Requirement Text |
| Input Method | Manual text entry in text area |
| Data Type | String (`str`) |
| Max Length | 1000 characters (no UI restriction; follows API limits) |
| Required/Optional | Required |
| Validation | Display error message and abort processing if empty string |

**Input Example**

```
Python-based web backend development. FastAPI and AWS experience required.
Team lead experience is a plus. Engagement expected for 3 months or more.
```

### 3.2 Output (Top-k Candidate Engineers + Rationale)

Output Format: List display in the Streamlit UI

| Field Name | Data Type | Description |
|------------|-----------|-------------|
| rank | int | Candidate rank (1-based) |
| engineer_name | str | Engineer name |
| skills | str | Comma-separated skill list (e.g. "Python, AWS, Docker") |
| similarity_score | float | Similarity score in the range [0.0, 1.0]. Computed as `max(0.0, 1 - cosine_distance)` where `cosine_distance` is returned by ChromaDB (configured with `hnsw:space="cosine"`). Mathematically equivalent to cosine similarity, with a 0.0 lower-bound clamp to absorb floating-point rounding. Higher = more similar. |
| matching_reason | str | Matching rationale generated by LLM |

**Output Example (UI display image)**

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Candidate 1: Ichiro Tanaka (Similarity: 0.87)
Skills: Python, FastAPI, AWS, Docker
Matching Rationale:
- Directly matches the FastAPI experience required, ready to contribute immediately.
- 3 years of AWS infrastructure experience adds value on the infrastructure side.
- Has lead engineer experience, satisfying the team lead requirement.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 4. Error Message List

| Code | Trigger Condition | Display Message |
|------|-------------------|-----------------|
| E001 | Input text is empty | "Please enter job requirements." |
| E002 | No data registered in ChromaDB | "Index not built. Please build it from the sidebar." |
| E003 | OpenAI API key not set | "API key is not configured. Please check your .env file." |
| E004 | OpenAI API call failed | "API call failed. Please wait a moment and try again." |
| E005 | engineers.json failed to load | "Failed to load data file. Please check the file path." |
| E006 | Search returned 0 results | "No matching engineers found. Please try different requirements." |

> Errors are displayed in red using the `st.error()` component.

## 5. API Interface (OpenAI API Call Specifications)

### 5.1 Embedding API (Text Vectorization)

| Item | Detail |
|------|--------|
| Endpoint | `POST https://api.openai.com/v1/embeddings` |
| Model | `text-embedding-3-small` |
| Input | Text string (job requirement or engineer profile) |
| Output | 1536-dimensional floating-point vector |
| Call Timing | During index build (all records from engineers.json, up to 20 calls) / During matching execution (input text, 1 call) |
| Error Handling | Catch `RateLimitError` and `AuthenticationError` → Convert to E004 / E003 |

### 5.2 Chat Completions API (Reason Generation)

| Item | Detail |
|------|--------|
| Endpoint | `POST https://api.openai.com/v1/chat/completions` |
| Model | `gpt-4o-mini` |
| max_tokens | 800 |
| temperature | 0.3 (prioritizing reproducibility) |
| Input (system) | Role instruction and output format specification for matching reason generation |
| Input (user) | Job requirement text + candidate engineer info (JSON format) |
| Output | Matching rationale text (in Japanese, bullet points) |
| Call Timing | During matching execution (up to k calls or 1 batch call for top-k candidates) |
