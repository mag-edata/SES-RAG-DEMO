# テスト設計書

**プロジェクト名:** SES-RAG-DEMO
**バージョン:** 1.0
**作成日:** 2026-03-20

---

## 1. テスト方針

- 「主要ユースケースがエンドツーエンドで動作すること」と「品質保証の厳密さ」の両立を目指す。
- 外部API(OpenAI)に依存するため、ユニットテストでは可能な限りモックを使用する。
- RAGの精度評価は定性評価を主軸とし、定量指標(Top-3 精度)は補助的に用いる。
- 開発者1名による手動テスト + pytest 実行で実施する。

---

## 2. テストスコープ

**対象**

- `rag/embedder.py`(Embedding生成およびDB登録ロジック)
- `rag/retriever.py`(類似度検索ロジック)
- `rag/chain.py`(プロンプト生成とLLM呼び出し)
- `app.py`(UI操作フロー)
- 結合テスト(全モジュールを通したエンドツーエンド動作)
- エラーケース(APIキー未設定、DB未構築 等)

**対象外**

- OpenAI API の内部動作(外部サービス)
- ChromaDB のコア実装
- Streamlit フレームワーク自体の動作

---

## 3. テストレベル定義

### ユニットテスト

| 項目 | 内容 |
|------|--------|
| 対象 | 各モジュールの個別関数 |
| 方法 | pytest + unittest.mock(外部API呼び出しはモック化) |
| タイミング | 各モジュール実装完了時 |
| 合格基準 | 全テストケースがPass |

### 結合テスト

| 項目 | 内容 |
|------|--------|
| 対象 | embedder → ChromaDB → retriever のパイプライン |
| 方法 | テスト用のインメモリChromaDBを利用し、実データで検証 |
| タイミング | 全モジュール実装完了後 |
| 合格基準 | 検索結果が正しく返却され、登録件数が一致 |

### システムテスト

| 項目 | 内容 |
|------|--------|
| 対象 | Streamlit UI を含むフル動作 |
| 方法 | ブラウザでの手動操作 |
| タイミング | デプロイ前の最終確認 |
| 合格基準 | 主要ユースケースが正常完了 |

---

## 4. テスト環境

**ローカル環境**

| 項目 | 内容 |
|------|--------|
| ChromaDB | pytest の `tmp_path` フィクスチャによる一時ディレクトリ(自動クリーンアップ) |
| OpenAI API | 実APIキーを使用(テスト費用予算に留意) |

**テスト構成**

- pytest の `tmp_path` フィクスチャでChromaDBの保存先を一時ディレクトリにリダイレクトし、テスト間の隔離を確保(`tests/test_retriever.py` の `chroma_dir` フィクスチャ参照)
- ユニットテストではOpenAI APIをモック化し、APIコストを発生させない

**Streamlit Cloud検証**

- デプロイ後、UIを手動操作してシステムテストを実施

## 5. テストデータ方針

### `jobs.json`(案件要件データ - 20件)

| テスト種別 | 用途 |
|-----------|-------|
| 結合テスト / システムテスト | 架空データをそのまま使用 |
| ユニットテスト | 使用しない。既存ユニットテストはすべてエンジニアデータかインライン辞書のみで動作する。 |

### `engineers.json`(エンジニアデータ - 20件)

| テスト種別 | 用途 |
|-----------|-------|
| ロード確認(E-01) | `load_engineers()` で20件をロードし件数を検証 |
| インデックス構築テスト(E-03, E-04) | テストコード内でインライン定義した3件のサンプル(`SAMPLE_ENGINEERS`)を使用(全20件は使わない) |
| 検索テスト(R-01〜R-04) | `SAMPLE_ENGINEERS`(3件)と固定のフェイクEmbedding(`[0.1] × 1536`)を使用。全候補が同一ベクトルとなるため、構造・件数・距離範囲のみ検証する(意味的ランキングは検証対象外) |
| 意味ランキング検証(R-05) | 保留。実OpenAI API呼び出しを行い、関連スキルを持つエンジニアが上位に出現することを確認する必要あり |
| ユニットテスト(モック) | テストコード内に2〜3件の固定サンプルをインライン定義(ファイルI/Oなし) |

**テストデータ原則**

- 実在企業名・実在個人名は使用しない
- テスト専用データが必要な場合は `engineers.json` に追記せず、別ファイルを作成する

---

## 6. 合否判定基準

### ユニットテスト

- すべての pytest ケースがPASS(エラー0件、失敗0件)
- 主要関数のキー分岐を最低限カバー

### 結合テスト

- ChromaDB に登録された件数と検索ヒット件数が一致する
- 類似度スコアが 0.0〜1.0 の範囲に収まる
- 返却件数が要求した top-k 件数と一致する

### システムテスト

- インデックス構築 → マッチング実行のフル動作がエラーなく完了する
- エラー発生時、UI上に適切なエラーメッセージが表示される
- 応答時間が目標値以内(検索:3秒、生成:10秒)

## 7. 評価指標

### Top-3 精度

| 項目 | 内容 |
|------|--------|
| 定義 | Top-3 内で人間が「適切」と判断した件数 / 3 |
| 測定方法 | 5件のテスト案件要件を人手でスコアリング |
| 目標値 | 平均0.6以上(5件中3件以上が適切と判断) |

### LLM生成理由の妥当性

| 項目 | 内容 |
|------|--------|
| 定義 | 生成されたマッチング理由が案件要件に根拠を持っているか |
| 測定方法 | 人手レビュー(0:不適切 / 1:部分的に適切 / 2:適切) |
| 目標値 | 平均スコア1.5以上 |

### 参考指標(将来の改善用)

| 指標 | 説明 |
|--------|-------------|
| MRR(Mean Reciprocal Rank) | 理想候補が何位に出現するかを測定 |
| Recall@k | 正解候補が Top-k 内に出現する割合 |

> デモ段階では定性評価を優先し、定量指標は参考値として扱う。
